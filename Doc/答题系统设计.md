# 智能答题系统 — 架构设计

## 一、核心数据模型

### 1.1 TagMastery（标签掌握度）

每个**独立标签**一条 SM-2 间隔重复记录，取代旧的 KnowledgePointMastery（标签组合）。

```
TagMastery
├── tag: string              # 标签名，即字典 key
├── SM-2 参数
│   ├── easeFactor           # 难度因子（初始 2.5，下限 1.3）
│   ├── interval             # 复习间隔（天，上限 180）
│   └── repetitions          # 连续正确次数
├── 统计
│   ├── totalAttempts         # 总答题次数
│   ├── avgScore              # 近 5 次均分
│   ├── recentScores[]        # 近 5 次分数
│   └── masteryLevel          # not_mastered | partially | mastered | expert
└── 时间戳
    ├── nextReviewAt          # 下次应复习时间
    └── lastReviewAt
```

**更新逻辑**: 一道题有 tags `[A, B, C]` 且得分 75 → 分别更新 `TagMastery['A']`、`TagMastery['B']`、`TagMastery['C']`。

### 1.2 StoreContext（向量库上下文）

出题管线使用的向量库封装，携带元数据：

```
StoreContext
├── store: LocalVectorStore   # 向量库实例（封装的搜索API）
├── storeId / name            # 标识与名称
├── role?: VectorStoreRole    # material | questions_no_answer | questions_with_answer
├── topicVocabulary?: string[] # 主题词表（构建时 LLM 提取）
└── embeddingConfig            # 该库对应的 embedding 配置（含 apiKey）
```

### 1.3 QuizQuestion（题目）

```
QuizQuestion
├── question / referenceAnswer / keyPoints[]
├── tags[]                    # LLM 出题时生成的知识标签
├── sourceChunkIds[]          # 来源向量块 ID（用于追溯）
├── type / difficulty / embedding?
└── usedCount / lastUsedAt    # 复用统计
```

## 二、出题管线

### 2.1 总体流程

```
用户选择向量库 + 参数
        │
        ▼
构建 StoreContext[]（含 topicVocabulary、embeddingConfig）
        │
        ▼
加载 TagMastery → 计算优先级 → 生成考生画像
        │
        ▼
┌───────────────────────────────────────────┐
│ 按配比分配四类出题任务（默认比例）         │
│                                           │
│  复习题 30%  → searchForTag()             │
│  薄弱强化 30% → searchForTag()            │
│  新知识 30%  → searchForNewTopic()        │
│  随机巩固 10% → searchStores(随机种子)    │
│                                           │
│  每类：搜索向量块 → 构建 prompt → LLM 生成 │
└───────────────────────────────────────────┘
        │
        ▼
SessionPlan { questions[], composition }
```

### 2.2 搜索策略

**不再全量加载向量块**，改为调用向量库封装 API `store.search(query, embeddingConfig, {topK})`。

| 阶段 | 搜索 query 来源 | 说明 |
|------|----------------|------|
| 复习题 | 该标签下最近历史题目文本 | 用题目文本比标签名更精准 |
| 薄弱强化 | 同上 | 优先搜索薄弱标签关联的历史题目 |
| 新知识 | 未覆盖的主题词 | embedding 匹配筛选未被标签覆盖的主题词 |
| 随机巩固 | 随机选取主题词或标签 | 多样性补充 |

每次搜索返回 topK=5 个向量块，多次搜索合并去重后取 5 个最相关块用于出题。

### 2.3 主题词-标签语义匹配

判断某个主题词是否已被现有标签覆盖：

1. **主方案**: 用 embedding 计算两者余弦相似度，≥0.75 视为覆盖
2. **降级方案**: embedding 失败时用字符串包含判断
3. **apiKey 来源**: 使用向量库自身的 embeddingConfig

## 三、向量库主题词表

在向量库 `buildIndex()` 的 Phase 3.95 生成：

1. 复用摘要生成时已采样的 20 个文本块
2. LLM 提取 20-30 个核心主题词（JSON 数组）
3. 存储到 `CollectionMeta.topicVocabulary`
4. 生成失败不阻塞构建流程

**主题词 vs 标签**:
- 主题词：向量库的静态属性，描述"库里有什么"
- 标签：题目的动态属性，描述"题考了什么"
- 两者通过语义匹配关联，不需要一一对应

## 四、评分管线

### 4.1 三级评分 + 校准

```
用户回答
    │
    ├─ Level 1: Cosine Similarity（答案 vs 参考答案 embedding）
    ├─ Level 2: Key Point Matching（逐个得分点 embedding 匹配）
    │           hit ≥ 0.75 / partial ≥ 0.58 / missed < 0.58
    └─ Level 3: LLM 四维评分
                关键点覆盖 40 + 准确性 25 + 完整性 20 + 清晰度 15
    │
    ▼
校准: LLM×50% + Cosine×25% + KeyPoint×25%
      + 低相似度封顶 (cosine<0.3 → cap 40, <0.4 → cap 60)
```

### 4.2 评分 Prompt 数据

评分 prompt 包含以下上下文供 LLM 参考：

```
┌─────────────────────────────┐
│ ① 题目 + 参考答案 + 得分点   │
├─────────────────────────────┤
│ ② 学生回答                   │
├─────────────────────────────┤
│ ③ 预评估数据                 │
│   ├ 语义相似度（Level 1）     │
│   └ 关键点初步匹配（Level 2） │
├─────────────────────────────┤
│ ④ 历史评判参考（同标签）      │
│   └ 近 2 道类似题的评判结果    │
│     ├ 题目 + 学生回答          │
│     ├ 评分 + 掌握等级          │
│     └ 得分/失分点              │
├─────────────────────────────┤
│ ⑤ 四维评分标准                │
│   关键点覆盖 40 / 准确性 25   │
│   完整性 20 / 表达清晰度 15   │
├─────────────────────────────┤
│ ⑥ 输出格式 (JSON)             │
└─────────────────────────────┘
```

### 4.3 语义相似度优化

Embedding 相似度容易偏高的问题通过以下方式缓解：

1. **分句匹配**：将回答拆分为句子级片段，逐片段与得分点计算相似度，取最大值
2. **阈值校准**：hit ≥ 0.72（严格），partial ≥ 0.55，missed < 0.55
3. **权重衰减**：校准阶段 cosine 仅占 25%，避免高相似度主导
4. **低分封顶**：cosine < 0.3 → 总分上限 40，cosine < 0.4 → 上限 60
5. **极端偏差修正**：LLM 与 cosine 差 > 40 分时取两者均值

## 五、动态 Prompt 体系

### 5.1 出题 Prompt

```
┌─────────────────────────┐
│ ① 角色定位               │  场景提示词 + 用户自定义提示词
├─────────────────────────┤
│ ② 数据源（按向量库分组）  │
│   ├ 向量库 A              │
│   │  ├ 摘要 + 主题词       │
│   │  ├ 块1: 文本 + 元信息   │
│   │  │  └ 关联标签历史      │
│   │  └ 块2: ...            │
│   └ 向量库 B ...            │
├─────────────────────────┤
│ ③ 考生画像               │  掌握率/薄弱标签/历史均分/总答题数
├─────────────────────────┤
│ ④ 阶段专属要求            │
│   ├ 复习题 (review): 换角度考察 + 1-2 个历史题目示例
│   ├ 薄弱强化 (weak): 降低难度 + 1-2 个历史题目示例
│   ├ 新知识 (new): 从基础切入，不依赖历史
│   └ 随机巩固 (random): 追求广度和多样性
├─────────────────────────┤
│ ⑤ 通用出题规则            │  题型/难度/去重/keyPoints 要求
├─────────────────────────┤
│ ⑥ 输出格式 (JSON)         │
└─────────────────────────┘
```

### 5.2 阶段策略（PHASE_STRATEGIES）

| 阶段 | 核心策略 | 历史示例 |
|------|---------|---------|
| review | 从不同角度或更高层次考察；重点攻击薄弱子知识点 | ✅ 给出 1-2 个历史题目+学生回答+得失分点 |
| weak | 适当降低难度；拆解知识点让学生逐步建立信心 | ✅ 给出 1-2 个历史题目+学生回答+得失分点 |
| new | 从基础概念入手，不预设学生已有知识 | ❌ 无历史 |
| random | 追求广度覆盖，选择不同子主题 | ❌ 无历史 |

## 六、存储架构

| 层级 | 存储 | 内容 |
|------|------|------|
| 热数据 | localStorage | UI 偏好、分类配置、集合元数据 |
| 温数据 | Electron saveAppData | TagMastery、题目缓存（500上限）、统计 |
| 冷数据 | Electron writeFile | 历史 Session 归档（按月） |

TagMastery 按分类（categoryId）隔离存储，支持旧 KnowledgePointMastery 自动迁移。

## 七、面试模式

与普通练习的区别：

- 逐题作答 → 即时评分 → LLM 判断是否追问（最多 3 轮）
- LLM 返回 `shouldFollowUp` + `followUpReason` + `interviewerComment`
- 所有追问完毕才进入下一题
- 全部题目完成后给出整体评测

## 八、掌握度仪表盘

### 8.1 数据统计

- 总练习/总答题/均分/连续天数（顶部卡片）
- 按标签统计：每标签答题次数、均分、掌握等级
- 分布环形图：四级掌握等级（未掌握/部分/已掌握/精通）的标签数量分布
- 分数分布柱状图：按 10 分段统计标签均分分布
- 趋势图：最近 N 次 session 的平均分走势

### 8.2 展示优先级

优先展示需要关注的标签（SM-2 优先级排序），每个标签显示：
- 掌握等级（颜色+图标）
- 记忆保持率（emoji）
- 均分 + 答题次数
- 复习建议原因
